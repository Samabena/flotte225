## 1. Introduction

### 1.1 Purpose

This Software Requirements Specification (SRS) defines the complete functional and non-functional requirements for **Flotte225**, a web-based vehicle fleet management system. It serves as the authoritative reference for design, development, testing, and validation. It is based on PRD v2.0, reverse-engineered from the existing codebase and validated by the Business Owner.

### 1.2 Scope

Flotte225 is a SaaS application targeting vehicle fleet owners in Côte d'Ivoire. It enables:

- Tracking fuel consumption and identifying anomalies
- Managing maintenance schedules and regulatory compliance deadlines
- Monitoring driver behavior via structured activity logs
- Generating AI-powered reports for owner decision-making
- Sending automated alerts and notifications (email, WhatsApp)
- Managing multiple fleet owners across subscription tiers via a super admin panel

### 1.3 Definitions & Acronyms

| Term | Definition |
|---|---|
| Owner | A registered user with the OWNER role who manages a fleet |
| Driver | A registered user with the DRIVER role who operates vehicles and submits fuel data |
| Super Admin | A privileged system-level administrator managing all accounts and system state |
| JWT | JSON Web Token — used for stateless authenticated sessions |
| FCFA | West African CFA Franc — the local currency used in Côte d'Ivoire |
| Webhook | An HTTP callback used to push fleet summaries to external tools |
| AI Report | A natural-language fleet summary generated by an LLM via OpenRouter |
| OTP | One-Time Password — a time-limited code for email-based authentication actions |

### 1.4 References

- PRD v2.0: `310326_Flotte225_PRD_v2.docx`
- Target market: Côte d'Ivoire (launch), French-speaking West Africa (expansion)
- Tech stack: FastAPI (Python 3.11+), PostgreSQL 15, Tailwind CSS, Docker Compose

- **SHALL** = mandatory requirement
- **SHOULD** = recommended, non-blocking
- **MAY** = optional or future scope
- `FR-XX` = Functional Requirement identifier
- `NFR-XX` = Non-Functional Requirement identifier

### 4.1 Authentication & Registration

#### 4.1.1 Registration

**FR-AUTH-01** The system SHALL allow any visitor to register a new account by providing:
- Full company name
- Full name
- Email address (unique in the system)
- Password
- Role selection: OWNER or DRIVER

**FR-AUTH-02** The system SHALL reject registration if the email address already exists in the system, returning a clear error message.

**FR-AUTH-03** The system SHALL hash all passwords using bcrypt before storage. Plaintext passwords SHALL NOT be stored or logged at any time.

**FR-AUTH-04** After successful registration, the system SHALL send a verification email containing an OTP code to the registered email address. The account SHALL remain inactive until the email is verified.

**FR-AUTH-05** The role assigned at registration SHALL be immutable. No user interface or API endpoint SHALL allow changing a user's role post-registration.

#### 4.1.2 Login

**FR-AUTH-06** The system SHALL authenticate users by validating email and password credentials.

**FR-AUTH-07** Upon successful login, the system SHALL issue a signed JWT (HS256) with a 24-hour expiration. The token SHALL be stored in `localStorage` on the client.

**FR-AUTH-08** The system SHALL redirect authenticated users to their role-appropriate dashboard automatically:
- OWNER → `dashboard-owner.html`
- DRIVER → `dashboard-driver.html`
- SUPER_ADMIN → `admin-dashboard.html`

**FR-AUTH-09** The system SHALL reject expired or tampered tokens with an HTTP 401 response, prompting the user to log in again.

#### 4.1.3 Password Reset

**FR-AUTH-10** The system SHALL provide a "Forgot password" flow accessible from the login page.

**FR-AUTH-11** The user SHALL enter their registered email address. If the email exists, the system SHALL send a 6-digit OTP code valid for **15 minutes**.

**FR-AUTH-12** The user SHALL enter the OTP code and, upon successful validation, be prompted to set a new password.

**FR-AUTH-13** A used or expired OTP SHALL be invalidated immediately and cannot be reused.

**FR-AUTH-14** The system SHALL NOT reveal whether an email exists in the system when OTP is requested (to prevent user enumeration). The response SHALL always be: *"If this email is registered, you will receive a reset code."*

### 4.2 Vehicle Management

*Access: OWNER only for write operations. Drivers have read access to assigned vehicles only.*

#### 4.2.1 Vehicle Creation

**FR-VEH-01** The OWNER SHALL be able to register a new vehicle with the following fields:

| Field | Required | Notes |
|---|---|---|
| Name / Identifier | Yes | Free label chosen by owner |
| Brand | Yes | e.g., Toyota, Renault |
| Model | Yes | e.g., Corolla, Clio |
| Year | No | Model year |
| License Plate | Yes | Unique within the system |
| VIN | No | Chassis serial number |
| Fuel Type | Yes | Essence, Diesel, GPL |
| Initial Mileage | Yes | Starting point for distance calculations |

**FR-VEH-02** The system SHALL reject vehicle creation if the license plate already exists in the system.

**FR-VEH-03** At the time of vehicle creation, the OWNER SHALL be able to assign one or more existing DRIVER-role users to the vehicle.

#### 4.2.2 Vehicle Modification

**FR-VEH-04** The OWNER SHALL be able to update all fields of an existing vehicle at any time.

#### 4.2.3 Vehicle Status — Pause

**FR-VEH-05** The OWNER SHALL be able to set a vehicle status to **Paused** (en panne / temporarily unavailable) without deleting it. A paused vehicle:
- Cannot receive new fuel entries
- Does not generate performance anomaly alerts
- Remains visible in all lists with a distinct "Paused" badge

**FR-VEH-06** The OWNER SHALL be able to restore a Paused vehicle to Active status at any time.

#### 4.2.4 Vehicle Archiving (Soft Delete)

**FR-VEH-07** The system SHALL NOT permanently delete vehicles. Instead, vehicles SHALL be **archived** (soft delete). Archived vehicles:
- Are hidden from all operational views (dashboard, fuel entry, active lists)
- Retain all historical data (fuel entries, maintenance records, activity logs)
- Are accessible in a dedicated "Archived Vehicles" section visible only to the OWNER
- Can be restored by the OWNER at any time

**FR-VEH-08** If a driver was active on a vehicle at the moment of archiving, their driving status SHALL be automatically reset to Inactive.

#### 4.2.5 Driver Assignment

**FR-VEH-09** The OWNER SHALL be able to add or remove drivers from a vehicle at any time.

**FR-VEH-10** Only users with the DRIVER role SHALL be assignable to vehicles.

**FR-VEH-11** A driver MAY be assigned to multiple vehicles simultaneously. However, a driver can only be **actively driving** one vehicle at a time (determined at activation by driver selection).

**FR-VEH-12** If a driver who is currently active (driving status = ON) is removed from a vehicle, the system SHALL automatically reset their driving status to Inactive.

#### 4.2.6 Visibility

**FR-VEH-13** An OWNER SHALL only see vehicles they registered. An OWNER SHALL NOT see other owners' vehicles.

**FR-VEH-14** A DRIVER SHALL only see vehicles to which they are currently assigned.

### 4.3 Fuel Entry Management

*Access: DRIVER creates entries. OWNER reads all entries for their fleet.*

#### 4.3.1 Submitting a Fuel Entry

**FR-FUEL-01** A DRIVER with Active driving status SHALL be able to submit a fuel entry with the following fields:

| Field | Required | Notes |
|---|---|---|
| Vehicle | Yes | Must be one of the driver's assigned vehicles |
| Date | Yes | Date of the refueling |
| Odometer reading (km) | Yes | Current mileage reading |
| Quantity (litres) | Yes | Volume of fuel added |
| Amount paid (FCFA) | Yes | Cost of the refueling |

**FR-FUEL-02** The system SHALL reject a fuel entry if:
- The driver is not Active (driving status = OFF)
- The driver is not assigned to the selected vehicle
- The odometer reading is not strictly greater than the previous reading for that vehicle

#### 4.3.2 Automatic Calculations

**FR-FUEL-03** Upon successful submission, the system SHALL automatically calculate and store:

| Indicator | Formula | Condition |
|---|---|---|
| Distance driven | Current odometer − previous odometer (or initial mileage for first entry) | Only if > 0 |
| Fuel consumption (L/100km) | (litres ÷ distance) × 100 | Only if distance > 0 |

#### 4.3.3 Edit & Delete (Driver)

**FR-FUEL-04** A DRIVER SHALL be able to edit or delete their own fuel entries **within 24 hours** of submission. After 24 hours, entries become immutable for drivers.

**FR-FUEL-05** Every edit and deletion SHALL generate an activity log entry capturing the before and after values (see §4.7).

**FR-FUEL-06** The OWNER SHALL have read-only access to all fuel entries for their fleet. The OWNER SHALL NOT modify or delete fuel entries.

### 4.4 Maintenance & Compliance

*Access: OWNER only.*

#### 4.4.1 Maintenance Records

**FR-MAINT-01** For each vehicle, the OWNER SHALL manage the following compliance indicators:

| Indicator | Description |
|---|---|
| Last oil change | Odometer reading (km) at the last oil change |
| Insurance expiry | Date the insurance contract expires |
| Technical inspection expiry | Date the technical inspection (visite technique) expires |

**FR-MAINT-02** All maintenance fields SHALL be optional. An owner may fill in none, some, or all fields per vehicle.

**FR-MAINT-03** If no maintenance record exists for a vehicle, the system SHALL automatically create an empty one on first access.

**FR-MAINT-04** The OWNER SHALL be able to update maintenance records at any time.

**FR-MAINT-05** The oil change alert SHALL be triggered by **mileage**, not by time. The system computes kilometers driven since the last recorded oil change odometer reading and fires alerts based on that value (see §4.8).

### 4.5 Owner Dashboard

*Access: OWNER only.*

**FR-DASH-OWN-01** The owner dashboard SHALL display the following sections in a single view:

#### Financial Indicators
- **Total fleet fuel spend**: sum of all fuel entry amounts across all vehicles
- **Spend per vehicle**: bar chart (alternating green/gold colors) showing each vehicle's total spend
- **Monthly trend**: line chart with gradient showing month-over-month spend evolution

#### Consumption Indicators
- **Average consumption per vehicle**: in L/100km
- **Consumption summary table**: list of all vehicles with their average consumption

#### Driver Status
- List of all drivers assigned to the owner's vehicles, each showing:
  - Name
  - Status (Active / Inactive)
  - If Active: which vehicle they are currently driving

#### Alerts & Anomalies
- Active maintenance/compliance alerts (classified by severity: Critical / Warning)
- Vehicles with detected performance anomalies

#### Compliance Deadlines
- Vehicles with insurance or inspection deadlines within the next 30 days
- Days remaining displayed

#### Visualizations
- **Donut chart**: spend distribution per vehicle
- **Circular gauge**: month-over-month spend growth rate
- **Operational gauge**: percentage of fleet that is currently operational

**FR-DASH-OWN-02** The operational fleet percentage SHALL be calculated as: *(vehicles with no critical alert AND not archived AND not paused) ÷ (total active vehicles) × 100*.

**FR-DASH-DRV-01** The driver dashboard SHALL display:

| Element | Description |
|---|---|

| Last 10 fuel entries | Date, vehicle, odometer, litres, amount |
| Driving status toggle | Button to activate/deactivate driving status |

**FR-DASH-DRV-02** When activating driving status, the DRIVER SHALL select which assigned vehicle they are currently driving.

**FR-DASH-DRV-03** The driver dashboard SHALL NOT display: fleet-wide analytics, other drivers' data, maintenance/compliance info, or performance alerts.

### 4.7 Activity Log (Audit Trail)

*Access: OWNER only.*

**FR-LOG-01** The system SHALL automatically record every create, update, and delete action performed by drivers on fuel entries.

**FR-LOG-02** Each log entry SHALL capture:

| Field | Description |
|---|---|
| Action type | CREATE, UPDATE, or DELETE |
| Timestamp | Date and time of the action (server time) |
| Driver | Who performed the action |
| Vehicle | The vehicle associated with the entry |
| Data snapshot | Full values at time of action; for UPDATE: both before and after values |

**FR-LOG-03** The OWNER SHALL be able to filter the activity log by:
- Driver
- Vehicle

**FR-LOG-04** The log SHALL be displayed in reverse chronological order (newest first).

### 4.8 Alerts & Anomaly Detection

*Access: OWNER only.*

#### 4.8.1 Maintenance Alerts

**FR-ALERT-01** The system SHALL generate the following alerts automatically:

| Type | Condition | Severity |
|---|---|---|
| Oil change | Km driven since last oil change > 500 km | Critical (red) |
| Oil change | Km driven since last oil change between 400 and 500 km | Warning (orange) |
| Insurance | Expiry date has passed | Critical (red) |
| Insurance | Expiry date within ≤ 30 days | Warning (orange) |
| Technical inspection | Expiry date has passed | Critical (red) |
| Technical inspection | Expiry date within ≤ 30 days | Warning (orange) |

**FR-ALERT-02** Alerts SHALL only be generated for vehicles that have the corresponding maintenance fields populated. If a field is empty, no alert is generated for it.

**FR-ALERT-03** Paused or archived vehicles SHALL NOT generate alerts.

#### 4.8.2 Performance Anomaly Detection

**FR-ALERT-04** The system SHALL flag a vehicle for **abnormal fuel consumption** when:
- The consumption of the latest fuel entry deviates by more than **20%** from the vehicle's historical average
- The vehicle has at least 2 entries with a calculated consumption value

**FR-ALERT-05** The system SHALL flag a vehicle for a **cost spike** when:
- Total fuel spend for the current month exceeds the previous month by more than **30%**
- Previous month data is available

**FR-ALERT-06** Anomalous vehicles SHALL appear in a dedicated section of the owner dashboard with the anomaly type(s) indicated.

### 4.9 Webhook Notifications

*Access: System admin only (configured via environment variable).*

**FR-WEBHOOK-01** The system SHALL support sending automated fleet summaries to an external HTTP endpoint (webhook URL) configured via the `WEBHOOK_URL` environment variable.

**FR-WEBHOOK-02** If `WEBHOOK_URL` is not set, webhook functionality SHALL be silently disabled with no error.

**FR-WEBHOOK-03** The webhook payload SHALL include:
- Owner identity (name, email)
- Period covered (since last notification)
- Alert summary (total, critical count, warning count, breakdown by type)
- Driver activity log delta (new creates, updates, deletions since last send)

**FR-WEBHOOK-04** Automatic webhook dispatch SHALL occur every 24 hours by default. The interval SHALL be configurable via the `WEBHOOK_INTERVAL_HOURS` environment variable.

**FR-WEBHOOK-05** The system SHALL record the timestamp and HTTP response status of the last webhook dispatch, queryable by the OWNER via the interface.

### 4.11 AI-Generated Reports

*Access: OWNER only. Subject to subscription plan (see §7).*

#### 4.11.1 Overview

**FR-AI-01** The system SHALL integrate with **OpenRouter** to generate natural-language fleet reports using a cost-effective LLM specialized in writing tasks (e.g., a capable model from the OpenRouter catalog).

**FR-AI-02** Reports SHALL be generated in **French**, as that is the sole supported language.

**FR-AI-03** The AI SHALL produce a report that is easy to understand by a non-technical fleet owner, including plain-language interpretation of:
- Overall fleet spend trend (is cost going up or down?)
- Best and worst performing vehicles (consumption, spend)
- Driver behavior summary (who is consistent, who has anomalies)
- Upcoming compliance deadlines requiring action
- Actionable recommendations (e.g., "Vehicle X needs an oil check soon")

#### 4.11.2 On-Demand Report

**FR-AI-04** The OWNER SHALL be able to trigger an AI report generation at any time by clicking a "Generate Report" button.

**FR-AI-05** The system SHALL pass a structured data snapshot of the owner's fleet (fuel entries, maintenance records, alert states, driver activity) to the LLM as context.

**FR-AI-06** The generated report SHALL be delivered to the owner's registered email address within 2 minutes of triggering.

**FR-AI-07** A loading indicator SHALL be displayed while the report is being generated. On success, a confirmation message SHALL inform the owner that the report was sent to their email.

#### 4.11.3 Scheduled Reports

**FR-AI-08** The OWNER SHALL be able to configure a recurring AI report schedule from the dashboard:
- Frequency: Weekly (every Monday) or Monthly (1st of each month)
- Enable / Disable toggle

**FR-AI-09** Scheduled reports SHALL be generated automatically by the backend scheduler (APScheduler) and sent to the owner's registered email.

**FR-AI-10** The owner SHALL be able to view the date and status (Sent / Failed) of the last scheduled report from the dashboard.

**FR-WA-04** The system SHALL send WhatsApp alerts specifically for:

**FR-WA-05** If WhatsApp API credentials are not configured in the environment, the feature SHALL be silently disabled.

### 4.13 Super Admin

*Access: SUPER_ADMIN role only. Accessible via a dedicated admin interface.*

#### 4.13.1 User & Account Management

**FR-ADMIN-01** The SUPER_ADMIN SHALL be able to view a paginated list of all registered users (owners and drivers), searchable by name and email.

**FR-ADMIN-02** The SUPER_ADMIN SHALL be able to **suspend** a user account, preventing them from logging in. A suspended user SHALL see a clear message explaining their account is suspended.

**FR-ADMIN-03** The SUPER_ADMIN SHALL be able to **reactivate** a suspended account.

**FR-ADMIN-04** The SUPER_ADMIN SHALL be able to **permanently delete** a user account. This action SHALL require a confirmation dialog and SHALL cascade-delete all associated data.

#### 4.13.2 Fleet & Vehicle Oversight

**FR-ADMIN-05** The SUPER_ADMIN SHALL be able to view any owner's vehicle list and fleet summary for support and auditing purposes.

#### 4.13.3 Subscription Management

**FR-ADMIN-06** The SUPER_ADMIN SHALL be able to view each owner's current subscription plan.

**FR-ADMIN-07** The SUPER_ADMIN SHALL be able to manually assign or change a subscription plan for any owner account (to handle manual payments, trials, or support exceptions).

**FR-ADMIN-08** The SUPER_ADMIN SHALL be able to activate or deactivate specific feature flags per owner account (overriding plan defaults).

#### 4.13.4 Platform Analytics

**FR-ADMIN-09** The SUPER_ADMIN SHALL have access to a platform-wide analytics dashboard showing:
- Total registered owners
- Total registered drivers
- Total vehicles in the system
- Total fuel entries logged
- Plan distribution (how many owners per plan)
- New registrations over time (chart)

#### 4.13.5 Super Admin Account Bootstrap

**FR-ADMIN-10** The first SUPER_ADMIN account SHALL be created via a secure database seed script run at deployment time, not through any public-facing endpoint.

### 4.14 Subscription Plans (SaaS)

*Access: Managed by SUPER_ADMIN. Feature enforcement applied at API level for each OWNER.*

**FR-PLAN-01** The system SHALL enforce three subscription tiers. Each owner account SHALL have exactly one active plan at all times. New accounts SHALL default to the **Starter** plan.

**FR-PLAN-02** Feature access SHALL be enforced server-side at the API layer — plan checks SHALL NOT rely solely on client-side logic.

**FR-PLAN-03** When an owner attempts to use a feature unavailable on their plan, the system SHALL display a clear upgrade prompt with a description of which plan unlocks that feature.

## 6. Non-Functional Requirements

### 6.1 Performance

**NFR-PERF-01** API response time SHALL be under **500ms** for 95% of requests under normal operating conditions.

**NFR-PERF-02** Database queries SHALL use indexed columns for all frequently filtered fields: `email`, `license_plate`, `vehicle_id`, `driver_id`, `date`.

**NFR-PERF-03** AI report generation (OpenRouter call + email delivery) SHALL complete within **2 minutes** under normal network conditions.

### 6.2 Security

**NFR-SEC-01** All passwords SHALL be hashed with **bcrypt** (minimum 10 rounds). Plaintext passwords SHALL never be stored, logged, or transmitted.

**NFR-SEC-02** JWTs SHALL be signed with **HS256** using a secret key stored in an environment variable (`SECRET_KEY`). Tokens expire after **24 hours**.

**NFR-SEC-03** Every protected API endpoint SHALL validate the JWT and enforce role-based access control server-side.

**NFR-SEC-04** Data isolation SHALL be enforced: an OWNER SHALL only access data belonging to their own account. No cross-owner data leakage is acceptable.

**NFR-SEC-05** CORS origins SHALL be restricted to trusted domains only, configured via the `CORS_ORIGINS` environment variable.

**NFR-SEC-06** OTP codes for password reset SHALL expire after **15 minutes** and be invalidated after first use.

**NFR-SEC-07** Sensitive environment variables (API keys, DB URLs, JWT secret) SHALL never be committed to source control.

### 6.3 Availability & Reliability

**NFR-AVAIL-01** The production environment (VPS + Docker Compose) SHALL use named Docker volumes to ensure database persistence across container restarts.

**NFR-AVAIL-02** The application SHOULD target **99.5% uptime** on the production VPS. Planned maintenance windows SHALL be communicated to active users.

**NFR-AVAIL-03** The staging environment on Render MAY have lower availability SLA and is not intended for end-user access.

### 6.4 Scalability

**NFR-SCALE-01** The backend architecture SHALL support horizontal scaling of the API layer (stateless JWT authentication allows multiple backend instances behind a load balancer).

**NFR-SCALE-02** Database migrations SHALL be managed via **Alembic** with versioned migration files, ensuring safe schema evolution.

### 6.5 Maintainability

**NFR-MAINT-01** The backend SHALL follow a clear module structure: routers, models, schemas, services, and dependencies separated by concern.

**NFR-MAINT-02** All scheduled tasks (webhook dispatch, scheduled AI reports) SHALL be managed by **APScheduler** with clear task identifiers and logging.

**NFR-MAINT-03** Background job failures (webhook, AI report) SHALL be caught, logged server-side, and surfaced to the owner as a status indicator in the UI.

### 6.6 Accessibility & Internationalization

**NFR-I18N-01** The application SHALL be served in **French** as the only language at launch.

**NFR-I18N-02** The existing `translations.js` mechanism SHALL be maintained in code to support future language additions without structural refactoring.

**NFR-I18N-03** The application SHALL be usable without a native app install, running fully in a modern mobile browser.

## 7. Subscription Plan Structure

The three tiers are designed so that the **Starter** plan provides genuine value to a small owner (1–2 vehicles), while the **Pro** and **Business** plans unlock the features that a growing fleet operation will need most.

### 7.1 Plan Overview

| Feature | Starter (Free) | Pro | Business |
|---|---|---|---|
| **Price (FCFA/month)** | 0 | 9 900 | 24 900 |
| **Vehicles** | Up to 2 | Up to 15 | Unlimited |
| **Drivers** | Up to 3 | Up to 20 | Unlimited |
| **Fuel entry & tracking** | ✅ | ✅ | ✅ |
| **Maintenance alerts** (insurance, inspection) | ✅ | ✅ | ✅ |
| **Oil change alert** (mileage-based) | ❌ | ✅ | ✅ |
| **Driver activity log** | Limited (last 20 entries) | Full | Full |
| **Performance anomaly detection** | ❌ | ✅ | ✅ |
| **Data export** (PDF + Excel) | ❌ | ✅ | ✅ |
| **AI report — On-demand** | ❌ | ✅ (5/month) | Unlimited |
| **AI report — Scheduled** (weekly/monthly) | ❌ | ❌ | ✅ |
| **WhatsApp notifications** | ❌ | ✅ | ✅ |
| **Email support** | Community | Standard | Priority |

### 7.2 Plan Enforcement Rules

**FR-PLAN-04** When a Starter owner reaches the vehicle limit (2), the system SHALL block vehicle creation and display an upgrade prompt.

**FR-PLAN-05** When a Starter owner reaches the driver limit (3), the system SHALL block driver registration prompts and display an upgrade prompt.

**FR-PLAN-06** On-demand AI report generation SHALL be rate-limited per plan:
- Starter: 0 reports/month
- Pro: 5 reports/month (counter resets on the 1st of each month)
- Business: unlimited

**FR-PLAN-07** A Starter owner attempting to access export, anomaly detection, WhatsApp, or AI reports SHALL see a locked-state UI with a clear upgrade call-to-action.

### 7.3 SaaS Simulation Mode

Since the payment gateway is not yet integrated, the system SHALL operate in **simulation mode**:

**FR-PLAN-08** The SUPER_ADMIN SHALL manually assign plan tiers to owner accounts via the admin panel.

**FR-PLAN-09** The system SHALL include a plan management data model (`subscription_plans`, `owner_subscriptions` tables) ready to be connected to a payment gateway in a future phase.

**FR-PLAN-10** The owner's profile page SHALL display their current plan and a "Upgrade" button that (in simulation mode) shows the plan comparison table and directs them to contact the administrator.
